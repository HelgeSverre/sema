## P0 — Critical

### [VM-1] `CallNative` native_id bounds check is `debug_assert!` only
- **Location:** `crates/sema-vm/src/vm.rs:1007-1020`
- **Confidence:** 100% (re-verified against source)
- **Why:** Line 1007 guards `native_id < self.native_fns.len()` with `debug_assert!`, which is
  compiled out in release builds. Line 1020 then does `self.native_fns[native_id].clone()` — a
  bounds-checked index that panics on a crafted `.semac` with an out-of-range `native_id`.
  `validate_chunk_bytecode` in `serialize.rs` has no `Op::CallNative` arm, so the bad index
  passes validation. Untrusted-bytecode denial of service.
- **Fix:** Add `Op::CallNative` validation in `validate_chunk_bytecode` (check
  `native_id < native_table_len`); replace the `debug_assert!` with a real bounds check that
  returns a `SemaError`.

## P1 — High

### [VM-2] `DUP` opcode does unchecked pointer deref on empty stack
- **Location:** `crates/sema-vm/src/vm.rs:790-793`
- **Confidence:** 95%
- **Why:** `unsafe { &*self.stack.as_ptr().add(self.stack.len() - 1) }` underflows
  `len() - 1` to `usize::MAX` when the stack is empty and reads far out of bounds (UB). No
  per-instruction stack-height verifier exists; a crafted `.semac` can declare `max_stack` that
  does not match actual stack effects and begin with a bare `DUP`. Same class as the documented
  `pop_unchecked` FIXME(C11) (`docs/limitations.md #32`), but `DUP` lacks even the debug_assert.
- **Fix:** Add a per-instruction stack-height verification pass to `validate_chunk_bytecode`;
  until then, guard `DUP` against an empty stack.

### [VM-3] Negative `i64` index wraps to `usize::MAX` in `NTH`
- **Location:** `crates/sema-vm/src/vm.rs:1622-1623`
- **Confidence:** 100%
- **Why:** `i as usize` where `i: i64 < 0` yields a huge positive value; `l.get(idx)` returns
  `None` and the error reports an astronomically large index instead of rejecting the negative.
  Logic/UX bug reachable from ordinary Sema code passing a negative index to `nth`.
- **Fix:** Check `i < 0` (return a type/bounds error) before the cast. See Pattern A in the index.

### [VM-4] Lambda rest-param omitted from optimizer shadow set → miscompilation
- **Location:** `crates/sema-vm/src/optimize.rs:134`
- **Confidence:** 92%
- **Why:** `optimize_inner` for `CoreExpr::Lambda` builds `param_names` from `def.params` only,
  skipping `def.rest`. If the rest param is named after a foldable builtin, constant folding
  fires inside the body even though the name is lexically shadowed. E.g. `(lambda (x . +) (+ 1 2))`
  folds the body to `Const(3)` instead of calling the rest-param `+`.
- **Fix:** Push the resolved `def.rest` name into `param_names` before `extend_shadowed`.

## P2 — Moderate

### [VM-5] Scheduler `mem::replace` is not panic-safe → lost tasks / silent deadlock
- **Location:** `crates/sema-vm/src/scheduler.rs:551`
- **Confidence:** 98% (impact conditional on a panic occurring)
- **Why:** A dummy `Scheduler` is swapped into the thread-local while the real one is used; it is
  re-extracted via `take_scheduler()` afterward. If task execution panics (e.g. an
  `unreachable!()` is hit) the take never runs, the empty shell is reinstalled, and all pending
  tasks/promises are silently dropped — callers deadlock.
- **Fix:** Restore the scheduler via a `Drop`/scope-guard so it is reinstalled even on unwind.

### [VM-6] `add_const` truncates constant-pool index to `u16`
- **Location:** `crates/sema-vm/src/emit.rs:45`
- **Confidence:** 85%
- **Why:** `self.chunk.consts.len() as u16` is unchecked. A unit with >65535 unique constants
  wraps the index to a previously-used slot; `Op::Const` then loads the wrong value (silent
  miscompilation). Extreme threshold, but fails silently.
- **Fix:** Guard `consts.len() > u16::MAX` → return `SemaError`; propagate from `emit_const`.

## P3 — Low

### [VM-7] `alloc_cache_slot` wraps inline-cache slots past 65535
- **Location:** `crates/sema-vm/src/compiler.rs:215`
- **Confidence:** 80%
- **Why:** `next_cache_slot.wrapping_add(1)` silently aliases after 65535 allocations; two
