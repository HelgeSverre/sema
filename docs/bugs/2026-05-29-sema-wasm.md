# sema-wasm — Audit Findings (2026-05-29)

**Status:** Open (unfixed) · **Scope:** `crates/sema-wasm/src/lib.rs` · Part of the
[2026-05-29 codebase audit](./2026-05-29-codebase-audit-index.md).
Line numbers as of commit `f609807` — re-confirm before fixing.

> In WASM a Rust panic becomes an unrecoverable trap that poisons the instance — so any
> reachable panic here is a hard crash of the sema.run playground.

## P1 — High

### [WASM-1] `timeout_ms as i32` truncation
- **Location:** `crates/sema-wasm/src/lib.rs:363`
- **Confidence:** 95%
- **Why:** `timeout_ms: u64` (from user JSON) cast `as i32` wraps for values > ~2.1e9 ms
  (~24 days) → negative/zero `setTimeout` delay → the abort controller fires immediately or at a
  wrong time, breaking/instantly-aborting the HTTP request. (Pattern A.)
- **Fix:** Clamp: `ms.min(i32::MAX as u64) as i32` (or `i32::try_from(ms).unwrap_or(i32::MAX)`).

## P2 — Moderate

### [WASM-2] `file/info` skips `normalize_path`, uses raw path as VFS key
- **Location:** `crates/sema-wasm/src/lib.rs:1453-1455`
- **Confidence:** 90%
- **Why:** Every other VFS function normalizes the path first; `file/info` looks up the raw
  string. `"foo"` (no leading `/`) or `"//foo"` reports "No such file or directory" even when the
  file exists under its canonical key, and it bypasses the `normalize_path` boundary entirely.
- **Fix:** Call `normalize_path(path)?` and use the canonical path for both lookups.

### [WASM-3] `escape_json` omits ASCII control characters → invalid JSON → silent NULL
- **Location:** `crates/sema-wasm/src/lib.rs:2765-2771`
- **Confidence:** 85%
- **Why:** `escape_json` only handles `\`, `"`, `\n`, `\r`, `\t`. Any other control char
  (U+0000–U+001F, e.g. from `(print "\x01")`) makes the hand-built JSON invalid; `JSON::parse`
  throws and `.unwrap_or(JsValue::NULL)` silently discards the entire result (value + output +
  error all become NULL). Affects ~20 call sites that share this pattern.
- **Fix:** Escape `\x00`–`\x1F` (minus the already-handled ones) as `\u00XX`, or build string
  values with `serde_json::to_string`.

### [WASM-4] `register_wasm_io` is a single ~1093-line function (V8 large-function crash risk)
- **Location:** `crates/sema-wasm/src/lib.rs:460-1553`
- **Confidence:** 80%
- **Why:** Directly relevant to the known V8 Turboshaft ARM64 large-function miscompilation that
  already caused a Chromium WASM crash for this project. This is by far the largest single
  function in the file and a prime Turboshaft optimization target.
- **Fix:** Split into logical registrars (`register_wasm_io_path`/`_vfs`/`_http`/…) each well under
  any compiler size threshold.

## Notes (not filed as findings)
- The ~21 `JSON::parse(...).unwrap_or(JsValue::NULL)` sites all share WASM-3's silent-data-loss
  risk; fixing `escape_json` once covers them.
- `closure.forget()` at `:365` leaks one small abort-controller closure per timed HTTP request —
  accepted WASM/no-GC trade-off, not a memory-safety bug.
- No bare `unwrap()`/index panics on JS-supplied data found outside the `escape_json` path.
