## P1 — High

### [EVAL-1] `unquote-splicing` in quasiquoted vector templates produces wrong output
- **Location:** `crates/sema-eval/src/special_forms.rs:979-984`
- **Confidence:** 95%
- **Why:** The vector branch of `expand_quasiquote` recurses on each item but never checks for
  `(unquote-splicing ...)`. `` `[1 ~@xs 2] `` yields a vector containing the literal two-element
  list `(unquote-splicing xs)` instead of splicing `xs`.
- **Fix:** Add the same splice-checking loop the list branch uses (lines 953-975) to the vector
  branch — extend rather than push when an item is `unquote-splicing`.

### [EVAL-2] `unquote` / `unquote-splicing` in quasiquoted map literals are ignored
- **Location:** `crates/sema-eval/src/special_forms.rs:985-987`
- **Confidence:** 95%
- **Why:** `expand_quasiquote` handles list and vector branches, then falls through to
  `Ok(val.clone())` for everything else — including `Value::Map`. `` `{:name ~name} `` returns
  `{:name (unquote name)}` verbatim regardless of `name`.
- **Fix:** Add a map branch that runs `expand_quasiquote` over each key/value and rebuilds the
  map (splicing in map values isn't meaningful; handle `unquote`).

## P2 — Moderate

### [EVAL-3] `eval_import` (tree-walker) bypasses the `FS_READ` sandbox check
- **Location:** `crates/sema-eval/src/special_forms.rs:1387`
- **Confidence:** 85%
- **Why:** `eval_load` (`:1619`) and the VM `__vm-import` delegate (`eval.rs:870`) both call
  `ctx.sandbox.check(Caps::FS_READ, ...)`, but `eval_import` has no sandbox check. Under a
  restricted sandbox (LSP, notebook) running with `--tw`, `(import "any/path.sema")` reads
  arbitrary files even when `FS_READ` is denied. (Caveat: VM is the default backend; this path
  requires the tree-walker.)
- **Fix:** Add `ctx.sandbox.check(Caps::FS_READ, "import")?` at the top of `eval_import`.
