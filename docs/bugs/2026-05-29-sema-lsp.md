# sema-lsp — Audit Findings (2026-05-29)

**Status:** Open (unfixed) · **Scope:** `crates/sema-lsp/src` · Part of the
[2026-05-29 codebase audit](./2026-05-29-codebase-audit-index.md).
Line numbers as of commit `f609807` — re-confirm before fixing.

The P1 here is a **cluster** of the same root cause: the lexer produces 1-indexed **char**
columns, while LSP `Position.character` is a 0-indexed **UTF-16 code-unit** offset. The two are
mixed without conversion, so everything breaks on lines containing non-ASCII (and especially
astral/emoji) characters. (Pattern B in the index.)

## P1 — High

### [LSP-1] char-index ↔ UTF-16 confusion across positions and ranges
- **Locations:**
  - `crates/sema-lsp/src/helpers.rs:104-114` — `span_to_range` emits char-index as UTF-16
    `character`. Affects **every** `Range`: diagnostics, go-to-def targets, **rename edits**,
    folding ranges, document highlights, selection ranges.
  - `crates/sema-lsp/src/helpers.rs:439-444` — `find_arg_positions_in_form` compares 1-indexed
    char col against a 0-indexed byte offset (then used as UTF-16 char in inlay hints).
  - `crates/sema-lsp/src/lib.rs:1472-1473, 1562-1563, 1619-1620, 1720-1721, 2675-2676` —
    `sema_col = position.character + 1` (UTF-16) passed to scope-tree queries that use char
    columns → wrong scope resolution (go-to-def/refs/rename/highlight return None or wrong target).
- **Confidence:** 88-92%
- **Why:** On any line with a multi-byte or astral character, ranges point at the wrong columns
  and scope queries misfire. The most damaging consequence: **rename can edit the wrong span and
  corrupt the user's source.**
- **Fix:** Add a char-index ↔ UTF-16 conversion utility (iterate the cached source line with
  `ch.len_utf16()`); apply it in `span_to_range`, in `find_arg_positions_in_form`, and when
  deriving `sema_col` from `position.character`.

## P2 — Moderate

### [LSP-2] Semantic-token `length`/`delta_start` computed in chars, not UTF-16
- **Location:** `crates/sema-lsp/src/lib.rs:2294` (and the `delta_start` at `:2317`)
- **Confidence:** 85%
- **Why:** `span.end_col - span.col` is a char count; LSP requires UTF-16 units. A symbol with
  non-ASCII chars gets a wrong token length, and `delta_start` shifts all later tokens on the line.
- **Fix:** Re-derive the UTF-16 length/delta from the symbol's chars via `len_utf16()`.

### [LSP-3] Always-true named-let guard can misclassify malformed `let`
- **Location:** `crates/sema-lsp/src/scope.rs:429`
- **Confidence:** 80% (latent)
- **Why:** `items[0].as_symbol().is_some()` is always true (head is always `"let"`). For a
  malformed `(let x 42)` (which the LSP still analyzes on broken files), `items[2]` is treated as
  a bindings list → wrong scope bindings until corrected.
- **Fix:** Drop the always-true clause and additionally require `items[2].as_list().is_some()`.
